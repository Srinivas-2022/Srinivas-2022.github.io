02 — Strategy One

Oracle RAC Rolling Upgrades

Oracle Real Application Clusters (RAC) is the foundation of zero-downtime patching. In a RAC environment, multiple database instances run on separate nodes against a shared storage layer. This architecture enables node-by-node patch application — while one node is being patched, the workload is automatically redistributed to surviving nodes.

    ORACLE RAC — ROLLING PATCH SEQUENCE
    ─────────────────────────────────────────────────────────

    SHARED STORAGE (ASM / ACFS)
    ┌──────────────────────────────────────────────────┐
    │  DATA FILES  ·  REDO LOGS  ·  CONTROL FILES     │
    └──────────────────────────────────────────────────┘
         │                 │                 │
    ┌────┴────┐        ┌────┴────┐        ┌────┴────┐
    │ NODE 1  │        │ NODE 2  │        │ NODE 3  │
    │  INST1  │        │  INST2  │        │  INST3  │
    │ ✔ LIVE  │        │ ✔ LIVE  │        │ ✔ LIVE  │
    └─────────┘        └─────────┘        └─────────┘

    STEP 1: Drain Node 1 connections (services migrate)
    ┌─────────────────────────────────────────────────────┐
    │ NODE 1: OFFLINE/PATCHING  │ NODE 2 ✔  │ NODE 3 ✔  │
    └─────────────────────────────────────────────────────┘

    STEP 2: Apply RU/PSU patch → validate → bring back
    ┌─────────────────────────────────────────────────────┐
    │ NODE 1 ✔ PATCHED          │ NODE 2: PATCHING   │ N3 ✔│
    └─────────────────────────────────────────────────────┘

    STEP 3: Final node patched — 100% cluster on new RU
    ┌─────────────────────────────────────────────────────┐
    │ NODE 1 ✔ PATCHED          │ NODE 2 ✔ PATCHED  │ N3 ✔│
    └─────────────────────────────────────────────────────┘

    ► ZERO application downtime throughout entire sequence
Oracle RAC Rolling Patch — Node-by-Node Sequence
🔄

Service Draining

Oracle Clusterware automatically migrates active services away from the node being patched using srvctl relocate service. In-flight transactions complete gracefully.

🛡️

Cluster Interconnect

The private network (10 GbE / InfiniBand) ensures cache coherency across nodes via Global Cache Services (GCS), maintaining data integrity during rolling operations.

⚡

Out-of-Place Patching

Modern OPatch installs to a new ORACLE_HOME, enabling rapid rollback. No in-place binary modification means the original home stays intact as a safety net.

💡

Pro tip: Always run opatchauto apply with -oh pointing to the new ORACLE_HOME, and verify with cluvfy comp patch before allowing connections back to a patched node. A failed checksum at this stage is far cheaper than a production incident post-cutover.

03 — Strategy Two

Oracle Data Guard Failover Mechanisms

For standalone (non-RAC) databases or as a complementary layer to RAC, Oracle Data Guard provides a primary-standby architecture that enables patching through a controlled switchover. The process patches the standby first, then executes a clean role transition — making the old standby the new primary without any data loss.

    DATA GUARD — ZERO-DOWNTIME PATCH SWITCHOVER
    ─────────────────────────────────────────────────────────

    PRE-PATCH STATE
    ┌──────────────────────────┐    Redo      ┌─────────────────────────┐
    │   PRIMARY (Site A)       │ ──────────► │  STANDBY (Site B)       │
    │   oracle_db1 · OPEN RW   │  (sync/MRP) │  oracle_db1_stby · MRP  │
    └──────────────────────────┘             └─────────────────────────┘
           ▲ App Traffic                             ▲ Reporting (optional)

    STEP 1: Patch STANDBY (primary continues serving traffic)
    ┌──────────────────────────┐             ┌─────────────────────────┐
    │   PRIMARY ✔ LIVE         │             │  STANDBY: MRP STOPPED   │
    │   Serving production     │             │  OPatch applying RU…    │
    └──────────────────────────┘             └─────────────────────────┘

    STEP 2: Switchover (zero data loss)
    ┌──────────────────────────┐             ┌─────────────────────────┐
    │  OLD PRIMARY → STANDBY   │  ◄────────  │  NEW PRIMARY (patched)  │
    │  Role transition: ~30s   │  Redo Ship  │  App traffic redirected │
    └──────────────────────────┘             └─────────────────────────┘

    STEP 3: Patch former primary (now standby)
    ┌──────────────────────────┐             ┌─────────────────────────┐
    │  STANDBY (patched) ✔     │  ◄────────  │  PRIMARY (patched) ✔    │
    │  MRP resumed, in sync    │             │  Full HA restored        │
    └──────────────────────────┘             └─────────────────────────┘

    ► Switchover window typically 15–45 seconds · RPO = 0 · RTO ≈ 30s
Data Guard Switchover-Based Patching — RPO=0, Minimal RTO
  • 1

    Cancel MRP on Standby

    Stop the Managed Recovery Process on the standby to allow OPatch to run safely. Redo logs continue to be shipped and archived on standby, just not applied.

  • 2

    Apply Patch to Standby ORACLE_HOME

    Execute opatchauto apply on the standby node. Verify patch application and catalog inventory before proceeding.

  • 3

    Restart MRP & Verify Sync

    Restart recovery and confirm standby is caught up: SELECT * FROM V$DATAGUARD_STATUS — confirm gap = 0 before initiating switchover.

  • 4

    Execute Switchover

    Initiate ALTER DATABASE COMMIT TO SWITCHOVER TO PRIMARY WITH SESSION SHUTDOWN. DGMGRL simplifies this to a single command with built-in safety checks.

  • 5

    Patch Former Primary (now Standby)

    With traffic now on the patched node, safely patch the original primary. Optionally switchover again to restore the original topology.

04 — Strategy Three

Ansible Automation — Eliminating Human Error at Scale

RAC rolling upgrades and Data Guard switchovers involve dozens of sequential commands across multiple nodes, with strict dependencies and validation checkpoints. Manual execution is error-prone and inconsistent. Ansible playbooks encode the entire patch lifecycle as idempotent, version-controlled automation — runnable repeatedly with identical results.

    ANSIBLE PATCH AUTOMATION — EXECUTION FLOW
    ─────────────────────────────────────────────────────────

    [ Control Node ]
         │
         ├── inventory/
         │      ├── rac_nodes.ini        (node1, node2, node3)
         │      └── dg_pairs.ini         (primary, standby)
         │
         ├── roles/
         │      ├── preflight/           ✔ pre-checks, diskspace, locks
         │      ├── stage_patch/         ✔ download, checksum, stage
         │      ├── drain_node/          ✔ srvctl relocate, wait
         │      ├── apply_opatch/        ✔ opatchauto apply -oh
         │      ├── validate_patch/      ✔ cluvfy, inventory verify
         │      ├── rejoin_node/         ✔ start instance, services
         │      └── postflight/          ✔ AWR snapshot, alert log scan
         │
         └── site.yml → patch_rac.yml → patch_dg.yml

    EXECUTION MATRIX (3-node RAC):
    ┌────────┬──────────┬───────────┬──────────┬────────────┐
    │ PHASE  │ NODE 1   │ NODE 2    │ NODE 3   │ CLUSTER    │
    ├────────┼──────────┼───────────┼──────────┼────────────┤
    │ P1     │ PATCHING │ SERVING   │ SERVING  │ 2/3 ACTIVE │
    │ P2     │ REJOINED │ PATCHING  │ SERVING  │ 2/3 ACTIVE │
    │ P3     │ REJOINED │ REJOINED  │ PATCHING │ 2/3 ACTIVE │
    │ DONE   │ PATCHED  │ PATCHED   │ PATCHED  │ 3/3 ACTIVE │
    └────────┴──────────┴───────────┴──────────┴────────────┘
Ansible Playbook Structure & RAC Patch Execution Matrix
Sample Task — Node Drain (roles/drain_node/tasks/main.yml)
---
- name: Relocate all services away from {{ target_node }}
  become: true
  become_user: oracle
  shell: |
    srvctl relocate service -db {{ db_name }} \
      -service {{ item }} \
      -oldinst {{ instance_name }} \
      -force
  loop: "{{ active_services }}"
  register: drain_result

- name: Wait for active sessions to drop to zero
  oracle_query:
    service: "{{ scan_service }}"
    sql: >
      SELECT COUNT(*) FROM V$SESSION
      WHERE INST_ID = {{ instance_id }}
      AND STATUS = 'ACTIVE'
      AND USERNAME IS NOT NULL
  register: session_count
  until: session_count.result[0][0] | int == 0
  retries: 30
  delay: 10

- name: Stop instance gracefully
  shell: srvctl stop instance -db {{ db_name }} -instance {{ instance_name }} -stopoption immediate
  become_user: oracle
🔁

Idempotency

Playbooks can be re-run safely after interruption. Each task checks state before acting — no double-patching, no partial states.

📋

Audit Trail

Every Ansible run produces a timestamped JSON log. Full traceability for change management, CAB approval, and post-incident review.

🔔

Pre/Post Validation

Automated AWR snapshot capture before and after, alert log scanning for ORA- errors, and cluster health verification gates protect against silent failures.

06 — Conclusion

Key Takeaways for Enterprise DBAs and Architects

Zero-downtime patching is not a single technology — it is a layered discipline that combines the right architecture (RAC + Data Guard), repeatable automation (Ansible), and disciplined process (pre/post gates, audit logs, staged rollouts). Each layer compensates for the limits of the others.

  • ✔

    Architecture first, tooling second

    No automation tool can substitute for a well-designed HA architecture. RAC and Data Guard must be operational and tested before being used as a patching mechanism.

  • ✔

    Out-of-place patching is non-negotiable

    Always maintain a rollback-capable ORACLE_HOME. The cost of staging is trivial compared to a failed patch with no recovery path at 2 AM.

  • ✔

    Automation is your runbook made executable

    If your runbook cannot be converted to Ansible without ambiguity, the runbook is incomplete. Automation forces precision that manual documentation never achieves.

  • ✔

    Patch more often, not less

    Fear-driven quarterly patching accumulates technical debt and vulnerability exposure. Zero-downtime capability enables monthly cadence — reducing both risk and regulatory gap.
